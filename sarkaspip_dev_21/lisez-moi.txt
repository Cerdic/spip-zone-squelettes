******************************************************
* Pour installer Sarka-SPIP pour la première fois... *
******************************************************

L'installation, est facile !

# Première étape : Installation du CMS SPIP (http://www.spip.net/fr)
> Télécharger la dernière version de SPIP (http://www.spip.net/fr_download)
> Décompacter l'archive téléchargée
> Envoyer, par FTP, le contenu du dossier à la racine du site ou dans un répertoire nommé spip (par exemple)
> Se connecter avec le navigateur sur le site, dans un dossier intitulé «ecrire» (adresse du type http://adresse_du_site/ecrire ou http://adresse_du_site/spip/ecrire)
> Suivre les instructions d'installation de SPIP (Pour aller plus loin : http://www.spip.net/fr_article402.html)

# Deuxième étape : Installation du squelette Sarka-SPIP
> Télécharger l'archive de Sarka-SPIP 1.9
> Décompacter l'archive téléchargée
> Envoyer, par FTP, le contenu du dossier à la racine du site ou dans le répertoire contenant SPIP

# Troisième étape : Affichage du nouveau graphisme
> Se rendre dans l'Espace privé
> Choisir «Interface complète»
> Effectuer un vidage du cache (Configuration >> Vider le cache)
> Vider le cache du navigateur puis rafraîchir ou actualiser l'affichage.

# Quatrième étape : Installation du plugin Accès Restreint par Groupes
Si vous voulez utiliser le plugin Accès Restreint par Groupes dont la gestion est intégrée au squelette depuis la version 1.9.1, il est nécessaire
de l'installer en suivant les instructions suivantes (voir aussi sous spip_contrib):
> Télécharger la dernière version du plugin à l'adresse http://www.spip-contrib.net/Le-plugin-acces-restreint-par
> Décompacter l'archive téléchargée
> Envoyer, par FTP, le contenu du dossier dans un répertoire /plugins à la racine du site
> Se rendre dans l'Espace privé
> Activer le plugin par le menu Configuration, Gestion des plugins


*************************************************
* Mise à jour à partir d'une version antérieure *
*************************************************

Si vous installez Sarka-SPIP 1.9 alors que vous utilisez déjà une version antérieure (1.1.x ou 1.2.x), vous avez du remarquer que beaucoup de fichiers ont disparu (.php3, par exemple) et que la structure des répertoires du squelette a changé. Dans ce cas, pour repartir d'un état propre, nous vous conseillons fortement de suivre la procédure complète  décrite ci-dessous: 

## Mise à jour de SPIP ##
    * Sauvegarder la base de données
    * Sauvegarder tout le contenu du site (en fait, seul le dossier /IMG est vraiment utile)
    * Procéder à la [mise à jour de SPIP->http://www.spip.net/fr_article1318.html]

## Mise à jour de Sarka-SPIP ##
    * Sauvegarder la base de données
    * Sauvegarder tous les fichiers et dossiers du squelette courant (utile dans le cas d'un squelette modifié ou personnalisé)
    * Télécharger l'archive de la nouvelle version de Sarka-SPIP
    * Décompacter l'archive téléchargée
    * Supprimer, par FTP, {{tous}} les anciens fichiers et dossiers du squelette (utile pour éviter de conserver des fichiers devenus obsolètes)
    * Envoyer, par FTP, le contenu du dossier à la racine du site ou dans le répertoire contenant SPIP
    * Effectuer un vidage du cache (Configuration >> Vider le cache)
    * Vider le cache du navigateur puis rafraîchir ou actualiser l'affichage (utile en particulier pour les styles)
    * Refaire, pas à pas, les éventuelles modifications ou personnalisations  sur les nouveaux fichiers...

## Important! ##
	* Attention cette version a été optimisée pour SPIP 1.9.x
	* Tous les fichiers .php3 ont été soit renommés (mes_fonctions, mes_options) soit supprimés.
	* Le fichier .htaccess n'est pas nécessaire (il est inutile de renommer le fichier htaccess.txt)

* Utiliser SPIP 1.9 en local *
******************************

Beaucoup d'entres vous débutent avec SPIP sous un environnement serveur en local avec des outils tels que EasyPHP (http://www.easyphp.org/) ou WAMP (http://www.wampserver.com/). C'est assez pratique car cela permet de faire des tests et de ne pas se préoccuper de l'hébergement même si cela cache souvent des problématiques liées à l'hébergeur justement.
# Pour aller plus loin : http://www.spip.net/fr_article1970.html
